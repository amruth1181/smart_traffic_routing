# 🏗️ Architecture — Smart Traffic Routing

This document describes the end-to-end architecture, the data flow, and the
engineering decisions behind each layer.

> **Note:** This is a personal, sanitized reimplementation using **synthetic**
> data (weather is pulled from the free public Open-Meteo API). It shares no
> proprietary code or data.

---

## 1. High-level view

The system has two connected tracks:

- **Real-time track** — ingest live-ish feeds and enrich them via stream joins.
- **Batch analytics + serving track** — land data in a warehouse, model it with
  SQL/dbt, orchestrate it with Airflow, and serve predictions + routing.

## 2. Layer-by-layer

### 2.1 Sources
- **Traffic** & **incidents**: synthetic, generated by `generate_mock_traffic.py`
  (~15,000 rows of Bay Area PeMS-style records).
- **Weather**: real, from the free public **Open-Meteo API** (no key required).

### 2.2 Ingestion — Kafka
Three producers publish JSON messages to three topics:

| Producer | Topic |
|---|---|
| `streaming/kafka_traffic_producer.py` | `traffic-data` |
| `streaming/kafka_weather_producer.py` | `weather-data` |
| `streaming/kafka_incident_producer.py` | `traffic-incidents` |

Kafka + Zookeeper run via `docker-compose`. This simulates real-time IoT feeds.

### 2.3 Stream processing — Spark Structured Streaming
`streaming/spark_mongodb_writer.py` (and `spark_combined_stream.py`):
1. Read all three Kafka topics.
2. Parse JSON with explicit schemas.
3. Convert timestamps and apply **watermarks** (30 min) to bound state.
4. Round lat/lon to 1 decimal to use as a spatial join key.
5. **Stream-to-stream join**: `traffic ⋈ weather ⋈ incidents`, matched by
   location and a ±15-minute time window.
6. Write the enriched records to **MongoDB** (`smart_traffic_db.traffic_weather_incidents`).

### 2.4 Warehouse load (Extract–Load) — NEW
`warehouse/load_to_postgres.py`:
- Extracts enriched records from **MongoDB** (falls back to the CSV if Mongo is
  empty/unreachable).
- Standardizes the schema, adds a `_loaded_at` timestamp.
- Loads into **Postgres** `raw.traffic_weather_incidents`.

Postgres is the warehouse for local dev. Because all connection settings are
**environment variables**, the same code points at **Cloud SQL / RDS / a managed
Postgres** in production; switching to **BigQuery/Snowflake** means changing the
dbt profile target only.

### 2.5 Transform + test (dbt) — NEW
The `dbt/` project implements a layered (medallion-style) warehouse:

```
raw.traffic_weather_incidents          (loaded by Python)
        │  dbt
        ▼
staging.stg_traffic_weather_incidents  (clean + typed, 1 row/observation, a VIEW)
        │  dbt
        ▼
marts.congestion_by_region             (avg speed/flow/incidents per region+freeway)
marts.hourly_traffic_summary           (hourly time-series rollup per region)
```

**Data-quality tests** (run with `dbt test`):
- `not_null` on keys (`event_time`, `region`, `speed`, KPIs).
- `accepted_values` on `severity` and `incident_type`.
- **Source freshness**: warns if newest load > 12h old, errors > 24h.

### 2.6 Orchestration — Airflow — NEW
`airflow/dags/smart_traffic_pipeline.py` runs daily:

```
generate_data ▶ load_to_warehouse ▶ dbt_run ▶ dbt_test
```

- Retries + retry delay on every task.
- `on_failure_callback` posts to **Slack** (if `SLACK_WEBHOOK_URL` set) or logs —
  i.e. **failure alerting**.

### 2.7 Serving — ML + routing
- `scripts/xgboost_model.py` trains an XGBoost model → `models/traffic_speed_model.pkl`.
- `A*/a_star_routing.py` + `dashboard/dashboard.py` build a NetworkX road graph,
  predict per-edge speed with the model, run **A\*** for the fastest route, and
  visualize it in **Streamlit**.

## 3. Cloud path (free tier)

| Local (dev) | Cloud (prod) |
|---|---|
| Postgres in Docker | GCP **Cloud SQL (Postgres)** or **BigQuery** |
| Kafka in Docker | **Confluent Cloud** free tier / MSK |
| Local Spark `local[*]` | **Dataproc** / Databricks |
| Airflow local | **Cloud Composer** / MWAA |
| Local files | **GCS / S3** buckets |

Only environment variables (and the dbt target) change — no application code.

## 4. Skills demonstrated (data engineering)

- **Python** ETL/EL scripting · **SQL** transformations & aggregations (dbt)

- **Warehouse** modeling (raw → staging → marts) + **dbt tests** & freshness
